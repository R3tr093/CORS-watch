Que signifie CORS

Cross-Origin Resource Sharing

Le protocole CORS (Cross-Origin Resource Sharing) est un protocole qui permet aux scripts s'exécutant sur un client de navigateur d'interagir avec des ressources d'une autre origine.

En quoi les CORS sont utiles ?

Cela est utile car, grâce à la règle de même origine suivie par XMLHttpRequest et fetch, JavaScript ne peut appeler que des URL ayant la même origine que l'emplacement d'exécution du script.

Par exemple, si une application JavaScript souhaite passer un appel AJAX vers une API s'exécutant sur un domaine différent, elle sera bloquée grâce à la stratégie de même origine.

La plupart du temps, un script exécuté dans le navigateur de l'utilisateur n'aurait besoin que d'accéder à des ressources de la même origine (pensez aux appels d'API au même backend qui servait le code JavaScript en premier lieu).

Par conséquent, le fait que JavaScript ne puisse pas normalement accéder à des ressources d'autres origines est une bonne chose pour la sécurité.

Identifier une réponse CORS

Lorsqu'un serveur a été configuré correctement pour permettre le partage de ressources d'origine croisée, des en-têtes spéciaux sont inclus.

Leur présence peut être utilisée pour déterminer qu’une demande prend en charge CORS.

Les navigateurs Web peuvent utiliser ces en-têtes pour déterminer si un appel XMLHttpRequest doit continuer ou non.

Quelques en-têtes peuvent être définis, mais le principal qui détermine qui peut accéder à une ressource est Access-Control-Allow-Origin.

Cet en-tête spécifie les origines pouvant accéder à la ressource. Par exemple, pour autoriser l'accès à partir de n'importe quelle origine, vous pouvez définir cet en-tête comme suit:

Access-Control-Allow-Origin: *

Ou il peut être réduit à une origine spécifique:

Access-Control-Allow-Origin: http://www.example.com

Comprendre les CORS

Il existe deux types de demandes CORS: les demandes "simples" et les demandes "de contrôle en amont", et c'est le navigateur qui détermine laquelle est utilisée. En tant que développeur, vous n'avez normalement pas à vous soucier de cela lorsque vous créez des demandes à envoyer à un serveur.

Cependant, il est possible que différents types de demandes apparaissent dans votre journal réseau et, dans la mesure où cela peut avoir un impact sur les performances de votre application, il peut être avantageux de savoir pourquoi et quand ces demandes sont envoyées.

Requêtes simples (GET, POST et HEAD)

Le navigateur considère que la demande est une demande "simple" lorsque la demande elle-même répond à un certain ensemble d'exigences:

  • L'une de ces méthodes est utilisée: GET, POST ou HEAD.
  • Un en-tête sécurisé CORS est utilisé
  • Lors de l'utilisation de l'en-tête Content-Type, seules les valeurs suivantes sont autorisées: application / x-www-form-urlencoded, multipart / form-data ou text / plain
  • Aucun écouteur d'événement n'est inscrit sur un objet XMLHttpRequestUpload.
  • Aucun objet ReadableStream n'est utilisé dans la demande
  • La demande est autorisée à continuer normalement si elle répond à ces critères et l'en-tête Access-Control-Allow-Origin est vérifié lorsque la réponse est renvoyée.
